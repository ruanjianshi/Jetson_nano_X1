# CLAUDE.md

> **适用范围**: 本项目 `code_design_sample`
> **强相关**: 所有代码必须同时遵守 `CODING_STANDARDS.md` (v3.0.0)

## 0. 项目核心硬约束 (Hard Facts)

**在执行任何编码任务前，必须将以下事实视为不可协商的前提：**

1. **命名体系**: 必须使用 `_t` (类型), `_impl_t` (实现类), `_base_t` (基类), `_priv_t` (私有实现), `_v` (变量后缀), `_p` (指针后缀)。**禁止**使用驼峰命名法或标准库风格的 `snake_case`。
2. **错误处理**: **禁止抛出异常**。公共接口必须返回 `xxx_err_t` 枚举。`try-catch` 仅限第三方库边界转换。
3. **资源管理**: 必须 RAII。裸指针默认**只表示非拥有关系**。禁止显式 `new/delete`。
4. **结构模式**: 复杂类必须使用 **Pimpl 惯用法** (`std::unique_ptr<Priv>`) 隐藏依赖。

## 1. Think Before Coding

**不要假设。不要隐藏困惑。明确展示权衡。**

在编写任何代码之前：
- 显式声明你对需求的假设。如果模糊不清，**必须提问**。
- 如果存在多种解释，列出它们，而不是默默选择一种。
- **映射检查**: 你生成的代码是否符合上述"核心硬约束"？如果不符，说明理由。
- 如果发现更简单的实现方式，指出它。在过度设计出现时提出异议。

## 2. Simplicity First

**用最少代码解决问题。不要投机。**

- 不要为单一用途的代码引入抽象层（基类/工厂）。
- 不要为未要求的"灵活性"或"可配置性"添加代码。
- 不要为不可能发生的场景编写错误处理（例如：检查 `this == nullptr`）。
- **50 行原则**: 如果你写了 200 行代码而 50 行就能解决，重写它。

> *结合规范*: 除非明确需要稳定 ABI 或隐藏重依赖，否则不要使用 Pimpl。

## 3. Surgical Changes

**只改必须改的。只清理自己弄乱的。**

编辑现有代码时：
- **禁止**顺手"优化"相邻代码、注释或格式。
- **禁止**重构没有损坏的模块。
- **严格遵守**现有文件的代码风格（即使它与规范有细微偏差，保持文件内一致性优先）。
- 如果发现不相关的死代码，**提及但不删除**。

**清理原则**: 只移除**本次改动**引入的未使用变量、头文件或函数。

## 4. Goal-Driven Execution

**定义成功标准。循环直到验证通过。**

将模糊的任务转化为可验证的目标：
- "添加校验" → "编写针对无效输入的测试用例，然后修改代码使其通过"。
- "修复 Bug" → "编写复现 Bug 的测试用例，然后修改代码使其通过"。
- "重构模块 X" → "确保重构前后所有相关单元测试结果一致"。

**多步骤任务**: 先陈述简要计划：
1. [修改 A] → 验证: [检查点 A]
2. [修改 B] → 验证: [检查点 B]

**AI 必须主动运行的门禁（对应规范 13.4）**:
- ✅ 编译通过 (`-Wall -Wextra -Werror`)
- ✅ `clang-format` 格式检查通过
- ✅ `clang-tidy` 关键项通过

## 5. Test-First Bug Fix

> *源自规范 2.4 和 13.1*

当你被要求修复一个 Bug 时，**必须**遵循以下闭环：
1. **写测试**: 在 `tests/` 下新增一个失败的测试用例，证明 Bug 存在。
2. **修代码**: 修改 `src/` 下的代码。
3. **验证**: 运行测试，确保**只有**你新增的那个测试从失败变为成功，且其他旧测试全绿。

## 6. 模块边界约束

- **禁止**跨模块包含 `src/` 内部头文件
- **禁止**上层模块依赖下层模块 (依赖方向: core → driver → protocol → utils)
- **内部实现**放 `detail/` 或 `internal/` 命名空间

## 7. C++/C 接口分离

- C++ 原生接口: `.h/.cpp` + 命名空间 + 类 + RAII
- C 兼容接口: `xxx_c.h` + `extern "C"` + 句柄式 API
- **禁止**在同一个头文件中混合两种风格

---

**这些准则有效运行的标志**:
- Diffs 中没有不必要的空白符或格式改动
- 没有因过度复杂而导致的反复重写
- 在错误发生前，AI 提出了澄清性问题
